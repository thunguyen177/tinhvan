Debug code trong nghiên cứu

Riêng khoản debug thì mình có rất nhiều kinh nghiệm vì code đến đâu là có bugs đến đấy :v Code bị lỗi không nhất thiết do code bị sai mà có thể do một trường hợp không được lường trước xảy ra. Ví dụ, variance gần như không bao giờ bằng 0 nhưng trong dữ liệu ảnh thì có một số pixel như ở góc luôn là hằng số nên có thể bằng 0. Do đó nếu trong code có tính log(variance) thì có thể ra NA.
Một lỗi mình hay gặp mà vẫn hay làm là copy & paste xong quên sửa số. Công nhận là thời gian tiết kiệm được chỉ bằng một lượng epsilon so với cái thời gian debug lỗi phát sinh @@
Nhiều khi code thấy kết quả này chắc không sai nhưng sao nó lại không bằng phương pháp khác nhỉ. Nhưng rất may, sau đó chạy trên dữ liệu lớn hơn thì thấy chậm quá đành phải tối ưu lại cái code. Nhờ tối ưu lại cái code sai mà mình phát hiện ra cái code cũ sai nốt. Sửa lại thì tèn ten ten ... kết quả thơm ngon :v Tại sao code chạy chậm? Code chạy chậm ngoài do việc sử dụng vòng lặp nhiều... thì còn có thể do cách chọn công thức cho tính toán. Ví dụ dùng \(\hat{Var(X)}=\frac{1}{n-1)(x_i-\bar{x})^2\) thì sẽ lâu hơn rất nhiều so với dùng \(\hat{Var(X)}=\frac{x_i^2-\frac{\bar{x}^2}{n}}{n-1)\). A good book on writing for computer science students is writing for computer science by Justin Zobel. In addition, below are some notes on writing and doing research:
  • When trying to modify a table, some gigantic equations, remember to keep the older version by copying that part to some other file or comment it out in Latex (command 'Ctrl + /' in Overleaf). If someone ask you to modify it, and you are not sure what that person want, draw a sketch of it and send it to that person to make sure of what you're doing, avoid your effort going to be a waste.
  • When submitting to a journal, you may want to check out, aside from guidelines that the journal/conference provides, you may want to go to google scholar to check out how recent papers from that journal/conference is written.
  • Overleaf issue: Overleaf often display the pdf file even if there arer errors. Therefore, check before submitting to journal
  • Not every journal/conference allows using extra packages. Avoid being rejected because of this!
  • Instead of downloading and processing data for your experiments from scratch, google some examples/tutorial that use the same dataset if possible. That's faster!
  • Video tutorial on how to use Google scholar and find latex citation format
      My research checklist: here are several things I've learned from my submitted paper:
      • Content: Stay consistent to avoid extra works during revision : I did experiments for $20\%-40\%$ missing data and then when I plot the running time for $20\%-50\%$ because I think I would looks nicer than $20\%-40\%$. Yet, a reviewer asked why, was there a specific reason? So, we added $50\%$ experiments to the report.
      • References For big revenues, reference format is also important:
        Make sure the following are avaiable for each reference: journal/conference name, year of publication.
        If the paper is published, cite the published version, not the archived one.
        References should be in the same format (it's not good to see some have year round brackets, other do not).
        Bibtex copied from Google Scholar may not be perfect. So, make sure to check the consistency.
      • Check grammar throughout (even for short sentences)
